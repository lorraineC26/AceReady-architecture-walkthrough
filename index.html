The Dashboard: Parallel Fetching in Action

When you open the AceReady dashboard, the app fetches two things at the same time — your past interviews AND available interviews from other users. This is not an accident — it is a deliberate performance choice.

TypeScript const [userInterviews, latestInterviews] = await Promise.all([ getInterviewsByUserId(user.id), getLatestInterviews({ userId: user.id }), ]);
Plain English
Run both database queries at the same time — do not wait for the first before starting the second...
Fetch all interviews this specific user created, sorted newest first...
Fetch finalized interviews from OTHER users (for practice), excluding this user own...
Both results arrive together — the page renders faster than if they ran one after the other.
⚡

This pattern — running independent operations simultaneously — is called parallelism. It is one of the most impactful performance techniques in web development. Here it cuts dashboard load time roughly in half. Promise.all is the tool that makes it happen — and it is used extensively in async Next.js server components.

One Brain, Two Bodies: Firebase Client vs. Admin

Firebase appears twice in AceReady — once for the browser, once for the server. They do different things and must never be confused.

🌐
firebase/client.ts
Runs in the browser. Used for sign-in, sign-up, and getting the user ID token from Google.
⚙️
firebase/admin.ts
Runs on the server only. Reads from Firestore, verifies session cookies, creates session tokens. Never exposed to the browser.
firebase/admin.ts import { cert, getApps, initializeApp } from "firebase-admin/app"; import { getAuth } from "firebase-admin/auth"; import { getFirestore } from "firebase-admin/firestore"; const initFirebaseAdmin = () => { const apps = getApps(); // make sure only one instance of the app is initialized if(!apps.length) { initializeApp( { credential: cert({ projectId: process.env.FIREBASE_PROJECT_ID, clientEmail: process.env.FIREBASE_CLIENT_EMAIL, privateKey: process.env.FIREBASE_PRIVATE_KEY?.replace(/\\n/g, "\n"), }) } ) } return { auth: getAuth(), db: getFirestore(), }; } export const { auth, db } = initFirebaseAdmin();
Plain English
Import the tools needed to start and use Firebase on the server.
Define a function that creates or reuses the Firebase Admin connection.
Check if Firebase has already been set up — Next.js can run this file many times per request.
Only initialize if there is no existing instance — this is the singleton pattern.
Connect using a service account credential (like a server-side API key).
These values come from environment variables — secret config not in the code.
Fix escaped newlines in the private key — a common formatting quirk when pasting keys into config.
Return both the Auth service and the Firestore database handle.
Export auth and db so other server files can use them with a single import.
💡

The singleton pattern (only initialize once) matters here because Next.js can re-run server modules on every request. Without this check, you would create hundreds of Firebase connections and quickly hit limits.

Getting You In the Door

How AceReady knows who you are — and keeps knowing it across every page visit.

The Hotel Key Card

When you check into a hotel, the front desk encodes your room permissions onto a key card. Every time you swipe the card at a door, the lock reads it instantly — without calling the front desk again.

AceReady works the same way. Firebase Auth is the front desk. Your session cookie is the key card. Every page you visit swipes that card.

🔑
Sign Up

Firebase Auth creates your identity and stores your profile in Firestore. You only do this once.

🎫
Sign In

Firebase verifies your password and issues a short-lived ID token (valid 1 hour).

🍪
Session Cookie

The server converts the ID token into a 7-day httpOnly cookie. This is what keeps you logged in.

Four Steps to Being Logged In

From password entry to a secure 7-day session — here is every step that runs when you click Sign In.

1
Enter email + password

The browser sends your credentials to Firebase Auth using the client-side SDK.

2
Firebase verifies your identity

Firebase checks your credentials against its secure user store and returns a short-lived ID token.

3
Server creates a session cookie

Your browser sends the ID token to the Next.js server. The server verifies it with Firebase Admin and creates a secure 7-day session cookie via a server action.

4
Cookie stored in your browser

The httpOnly cookie is saved. Your browser will send it automatically on every future request — enabling persistent authentication for 7 days without you logging in again.

💡

Why does Firebase issue an ID token AND a session cookie? The ID token is for the browser (client-side verification). The session cookie is for the server (persistent authentication). Two different tools for two different jobs.

setSessionCookie — The Security Layer

This server function runs once after you sign in. It takes the ID token from Firebase and bakes it into a secure httpOnly cookie.

lib/actions/auth.action.ts export async function setSessionCookie(idToken: string) { const cookieStore = await cookies(); const sessionCookie = await auth.createSessionCookie(idToken, { expiresIn: ONE_WEEK * 1000, // 7 days in milliseconds }); // Set the session cookie in the response headers cookieStore.set("session", sessionCookie, { maxAge: ONE_WEEK, httpOnly: true, // Cookie is only accessible via HTTPS secure: process.env.NODE_ENV === "production", // Use secure cookies in production path: "/", // Cookie is valid for the entire site sameSite: "lax", // CSRF protection }); }
Plain English
A server function that takes the ID token Firebase gave the browser...
Get access to the response cookie jar...
Ask Firebase Admin to create a long-lived session cookie from the short ID token...
Make it expire in 7 days (Firebase needs milliseconds, so multiply)...
Store the cookie in the browser with these security settings...
httpOnly: the browser cannot read this cookie with JavaScript — attackers cannot steal it...
secure: only send over HTTPS in production (not over plain HTTP)...
path: this cookie applies to every page, not just one route...
sameSite lax: blocks the most common cross-site attack (CSRF) while allowing normal links...
⚠️

If httpOnly is ever removed from this cookie, malicious JavaScript on the page could steal the session. This is called XSS (Cross-Site Scripting). httpOnly is a one-line defense against it.

Your Voice, Live on the Wire

How spoken words become structured data — and how the Agent component listens for it all.

The Court Reporter

During a trial, the court reporter does not wait for the judge to ask "what did the witness say?" — they transcribe everything the moment it is spoken, in real-time.

Vapi is AceReady's court reporter. It listens to the voice call, transcribes every word, and fires events your code can react to. The Agent component subscribes to those events using the Observer pattern.

⚪
INACTIVE

Default state. The call button shows "Call". No Vapi connection exists.

🔄
CONNECTING

User clicked the button. Vapi is establishing a WebSocket connection. The button shows "..."

🟢
ACTIVE

Call is live. The AI interviewer is asking questions. Vapi is performing real-time transcription of every word.

✅
FINISHED

Call ended. The Agent triggers feedback generation (or redirects to dashboard if it was a setup call).

⚡

The Observer Pattern: instead of checking every millisecond "did the call start?", the code says: call me WHEN it starts. This is how all modern real-time apps work — from chat apps to live sports scores.

The Event Listener Setup

When the Agent component mounts, it registers 6 event listeners with Vapi. Each listener waits for a specific thing to happen during the call.

components/Agent.tsx useEffect(() => { const onCallStart = () => setCallStatus(CallStatus.ACTIVE); const onCallEnd = () => setCallStatus(CallStatus.FINISHED); const onMessage = (message: Message) => { if (message.type === "transcript" && message.transcriptType === "final") { const newMessage = { role: message.role, content: message.transcript }; setMessages((prev) => [...prev, newMessage]); } }; const onSpeechStart = () => setIsSpeaking(true); const onSpeechEnd = () => setIsSpeaking(false); const onError = (error: Error) => console.log("Call error:", error); vapi.on("call-start", onCallStart); vapi.on("call-end", onCallEnd); vapi.on("message", onMessage); vapi.on("speech-start", onSpeechStart); vapi.on("speech-end", onSpeechEnd); vapi.on("error", onError); return () => { vapi.off("call-start", onCallStart); vapi.off("call-end", onCallEnd); vapi.off("message", onMessage); vapi.off("speech-start", onSpeechStart); vapi.off("speech-end", onSpeechEnd); vapi.off("error", onError); }; }, []);
Plain English
Run this setup code once when the component first appears on screen (useEffect with empty dependency array)...
When the call connects: mark the call as ACTIVE...
When the call disconnects: mark the call as FINISHED...
When a speech message arrives: only process it if it is a finalized transcript (not a partial draft)...
Package the message as an object with the speaker role and their words...
Add the new message to the running list of all transcript messages...
When someone starts speaking: set the speaking indicator to true (shows animation)...
When speaking stops: set the speaking indicator to false...
Register all 6 listeners with Vapi, then return a cleanup function that removes them all when the component is removed from the page...
💡

The cleanup return function is critical. Without it, every time you navigate away and back, a NEW set of listeners would be added — on top of the old ones. You would end up with duplicate messages and ghost transcriptions.

The AI That Grades You

How a voice transcript becomes a structured scorecard — and the Gemini prompt engineering behind it.

The Customs Officer with a Checklist

When you cross a border, the customs officer does not accept a verbal story — they need your passport filled out in a specific format. Gemini works the same way: you do not send it a transcript and hope for a nice answer. You give it a strict form (a schema) it must fill in exactly.

Gemini has two jobs in AceReady. Both use the same language model, but in completely different ways.

📝
Job 1: Generate Questions

Before the interview, Gemini receives the role, level, and tech stack. It returns a JSON array of interview questions — formatted for voice (no special characters).

🎯
Job 2: Grade the Transcript

After the interview, Gemini receives the full conversation transcript. It returns a structured output JSON scorecard across 5 categories.

💡

The same AI model doing two completely different things based on two different prompts. The prompt IS the program. This is why prompt engineering is a real skill — small changes to the prompt dramatically change the output.

Telling Gemini What to Ask

The question generator uses a simple text prompt with placeholders filled in at runtime. Notice the special instruction at the end — interview questions must be voice-friendly.

app/api/vapi/generate/route.ts const { text: questions } = await generateText({ model: google("gemini-2.5-flash-lite"), prompt: `Prepare questions for a job interview. The job role is ${role}. The job experience level is ${level}. The tech stack used in the job is: ${techstack}. The focus between behavioral and technical questions should lean towards: ${type}. The amount of questions required is: ${amount}. Please return only the questions, without any additional text. The questions are going to be read by a voice assistant so do not use "/" or "*" or any other special characters which might break the voice assistant. Return the questions formatted like this: ["Question 1", "Question 2", "Question 3"] Thank you! <3 `, });
Plain English
Call Gemini and wait for the response...
Use the lightweight flash model — fast and cheap for simple text tasks...
Build the prompt by filling in the user choices: role and level...
Continue filling in the tech stack and question type (behavioral vs technical)...
Specify how many questions to generate...
Tell Gemini to skip explanatory text — just return the raw questions...
Voice assistants cannot say slash or asterisk — forbid them in the output...
Require the questions as a JSON array so the code can parse them reliably...
💡

The "Thank you! <3" at the end of the prompt is not just politeness — research shows that politeness markers in prompts can slightly improve AI output quality. It also makes the codebase more charming.
